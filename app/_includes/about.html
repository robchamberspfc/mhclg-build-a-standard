About

Aim

This prototype aims to demonstrate an idea for how we communicate and how users interact with data standards.

The basic idea is to allow a user to build out their data model a few different ways. Either by selecting individual fields/codelists or by working from a prebuilt 'template' (e.g. a use case from HACT)

Benefits

Having a clearly articulated data model will allow better communication and understanding of what exists.

For example in the case where a HP want to change supplier. We have heard that this task is complicated further by the lack of understanding of the current data model. Helping HPs have one should help them understand their own data in future.

Future ideas/limitations

Not everything is currently possible in this prototype. The big 2 options missing are around exactly how a user would supply custom fields/codelists and the ability to dig a level deeper into the information to see more about a specific fields or codelist.

  • Could allow users to search by a value they need and see what codelists contain it
  • Could validate supplied codelist to access if there is something existing that may meet requirements - nudging someone towards something standard.
  • Could integrate with Data Futurists 'index' tool to provide a bridge between current data.

Notes

Prototype uses local browser storage to remember your basket - meaning it should persist until cleared with the button or by deleting the browser data.
